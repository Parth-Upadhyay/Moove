# Moove (Kinetiq)
AI-Assisted Physical Therapy Monitoring and Clinical Triage Platform

---

## Overview

Moove is an Android-based Physical Therapy (PT) management system that integrates computer vision, structured clinical rules, and AI-generated summaries to assist healthcare providers in monitoring patient recovery.

The platform enables real-time tracking of patient exercises, automated Range of Motion (ROM) analysis, and clinically relevant triage insights through a hybrid decision system.

---

## Key Features

- Real-time pose tracking using CameraX and ML Kit  
- Automated Range of Motion (ROM) measurement  
- Session-level analytics (ROM, duration, exercise type)  
- Pain tracking (pre-session and post-session)  
- Hybrid AI-based clinical triage  
- Real-time doctor dashboard with alerts  
- Scalable cloud backend using Firebase

## System Architecture

The system follows a modular MVVM architecture with clear separation of concerns:

- UI Layer (Jetpack Compose)  
- ViewModel Layer (State Management)  
- Repository Layer (Data + AI Logic)  
- Data Layer (Firebase Firestore)  

---

## Patient Session Tracking

The application uses CameraX and ML Kit Pose Detection to monitor exercises in real time.

### Data Captured
- Peak Range of Motion (degrees)  
- Exercise duration  
- Exercise type  

### Patient Feedback
- Pre-session pain score (0–10)  
- Post-session pain score (0–10)  

### Data Storage
- Stored in Firestore under the `sessions` collection  
- Represented as `SessionResult` objects  

---

## AI Analysis Architecture (Hybrid Approach)

Moove uses a hybrid architecture combining a deterministic rules engine with a Large Language Model (LLM).

### 1. Rules-Based Triage (Primary Decision Layer)

Clinical priority is determined using explicit rules to ensure reliability and explainability.

- HIGH Priority:
  - Post-session pain ≥ 8  
  - ROM decline > 10%  
  - Severe pain trends within 72 hours  

- MEDIUM Priority:
  - Pain score > 4  
  - ROM improvement < 5% over two weeks  

- LOW Priority:
  - Consistent ROM improvement  
  - Stable pain scores < 4  

This ensures all clinical decisions are deterministic and auditable.

---

### 2. LLM-Based Summary Generation (Assistive Layer)

A Large Language Model is used to generate structured summaries.

- Converts numerical data into clinical narratives  
- Highlights trends such as regression or improvement  
- Produces concise summaries for clinicians  
- Enforces JSON-only output format  

Example output:

```json
{
  "priority": "HIGH",
  "message": "The patient shows a significant decline in range of motion accompanied by high pain levels, indicating the need for immediate clinical attention."
}
```

## 3. Processing Flow
The data pipeline follows a sequential flow from capture to clinical visualization:

1. **[Session Data]** *Captured via CameraX/ML Kit and user input.* ↓  
2. **[Rules Engine → Priority Classification]** *Deterministic logic categorizes the session (High/Medium/Low).* ↓  
3. **[LLM → Summary Generation]** *Llama-3.1 translates metrics into a natural language clinical narrative.* ↓  
4. **[Doctor Dashboard]** *Insights are delivered to the clinician's interface.*

---

## Doctor Dashboard
The clinician interface is designed for high-stakes monitoring and efficient triage:

* **Real-time updates:** Utilizes Firestore snapshot listeners to provide instant feedback as patients complete sessions.
* **Patient-wise AI alerts:** Automatically sorts and highlights patients based on priority levels (HIGH, MEDIUM, LOW).
* **Batch processing:** Efficiently manages and synchronizes data for all patients assigned to a specific doctor.
* **Regex-based sanitization:** Ensures structured parsing of AI-generated JSON to prevent UI crashes from malformed LLM responses.

---

## Data Models
The system architecture relies on three core data entities:

### Patient
* **Demographic information:** Name, age, clinical history.
* **Unique identifiers:** Firebase UID and clinical reference IDs.

### SessionResult
* **Exercise metrics:** Peak Range of Motion (ROM), duration, and repetition count.
* **Pain log array:** Longitudinal tracking of pain scores within a session.
* **Results object:** The final calculated output of the session's physical performance.

### AiAlert
* **Timestamped summary:** The date-marked narrative generated by the AI.
* **Priority level:** The clinical urgency status (HIGH, MEDIUM, LOW).

## Security and Configuration
Security is integrated at the code and infrastructure levels:

* **API Management:** Keys are stored securely using `BuildConfig` and `secrets.properties` to keep sensitive credentials out of version control.
* **Rate Limiting:** Implemented at the repository layer to prevent API overuse and manage operational costs.
* **Firestore Security Rules:** Granular access control policies that restrict data visibility strictly to authorized `doctorId` and associated `patient_id`.

---

## Design Rationale
The platform’s architecture is driven by clinical safety and usability:

* **Deterministic Safety:** Rules-based triage ensures that clinical decisions are consistent and not subject to LLM "hallucinations."
* **Interpretability:** The LLM bridges the gap between raw data and human understanding, making summaries readable at a glance.
* **Hybrid Flexibility:** This approach balances the reliability of traditional programming with the linguistic flexibility of modern AI.
* **Future-Ready:** The data schema is built to accommodate future unstructured inputs like text notes or voice memos.
